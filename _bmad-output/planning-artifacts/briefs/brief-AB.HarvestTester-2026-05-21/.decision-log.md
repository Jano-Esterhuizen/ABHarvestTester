# Decision Log — TestForge Product Brief

## 2026-05-21

### Decision: Project-Agnostic Scope
- **Context**: Originally scoped to Harvest (time-tracking) testing. User explicitly wants agnostic tool.
- **Decision**: Product targets any smaller web project with API/UI. Harvest is the demo target, not the product scope.
- **Rationale**: Reusability beyond competition; any Drawbridge team can adopt.

### Decision: Fast Path
- **Context**: User has evening to prepare for tomorrow's competition.
- **Decision**: Draft with [ASSUMPTION] tags, iterate quickly.
- **Rationale**: Time constraint is the primary blocker.

### Decision: Target Complexity
- **Context**: User specified "smaller, less complicated projects" — not enterprise-scale.
- **Decision**: v1 explicitly targets small-to-medium web projects. Large/complex repos are out of scope.
- **Rationale**: Achievable in timeframe; avoids over-engineering.

### Decision: Competition-First, Tool-Second
- **Context**: Primary trigger is conference competition; secondary goal is lasting utility.
- **Decision**: Optimize for demo impact AND genuine functionality. Not a throwaway.
- **Rationale**: Win the competition AND have something real to show after.

### Decision: Configuration Model
- **Context**: User wants minimal but sufficient config. Needs repo, hosted URL, and role credentials.
- **Decision**: Three inputs required: (1) repo path/URL, (2) live hosted URL, (3) credentials JSON with role-based accounts.
- **Rationale**: Repo Reader discovers roles/permissions from code; credentials file lets the system actually log in as each role during testing.

### Decision: Auth IS In Scope
- **Context**: User confirmed multi-role testing is core to the value prop.
- **Decision**: System tests with provided credentials across all configured roles. Complex OAuth/SSO discovery is out, but using provided creds to test role-based access is in.
- **Rationale**: Role-based testing (user vs admin vs super-admin) is a key differentiator for the demo.

### Decision: Repo Reader Discovers Roles
- **Context**: User wants the Repo Reader agent to identify the application's role/permission model from code.
- **Decision**: Repo Reader outputs include role/permission structure as part of its context document. This feeds into test writers to generate role-appropriate scenarios.
- **Rationale**: Automation of role discovery reduces config burden and impresses judges.
